# Task 02 — Конфигурация и логирование

## Цель

Реализовать пакет `internal/config`: загрузка YAML-конфига, строгая валидация, получение токена из переменной окружения. Настроить `log/slog` с уровнем из конфига.

## Контекст

- Спецификация: [../spec.md](../spec.md), разделы 3 (формат и валидация), 7 (логирование), 8 (коды выхода).
- Зависимость: task-01 выполнена.

## Шаги

1. Добавить `gopkg.in/yaml.v3`.
2. Структуры `Config`, `HomeAssistant`, `Entity`, `Notifications`, `Logging` — поля строго по разделу 3.1 спецификации.
3. Загрузка с `yaml.Decoder.KnownFields(true)` — неизвестные ключи являются ошибкой.
4. Валидация по разделу 3.2; все сообщения об ошибках — на русском, с указанием поля и причины.
5. Чтение токена: `os.Getenv(cfg.HomeAssistant.TokenEnv)`, пустое значение — ошибка конфигурации.
6. В `main.go`: загрузка конфига, при ошибке — сообщение в stderr и выход с кодом 2; инициализация slog с уровнем из `logging.level`.
7. Положить пример конфига `config.example.yaml` в корень (без секретов, с комментариями по-русски).

## Критерии приёмки

- [ ] Валидный конфиг из раздела 3.1 спецификации загружается без ошибок.
- [ ] Юнит-тесты покрывают: валидный конфиг; неизвестный ключ; отсутствие `entities`; сущность одновременно со `states` и `template`; сущность без обоих; невалидная схема URL; пустая переменная токена.
- [ ] Запуск без конфига завершает процесс кодом 2 с внятным русским сообщением.
- [ ] Токен нигде не логируется (проверить по коду: значение токена не попадает в аргументы slog).
