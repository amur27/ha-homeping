# Task 04 — Нативные уведомления

## Цель

Реализовать пакет `internal/notify`: показ нативного уведомления ОС, преобразование события сущности в заголовок/текст по конфигу, троттлинг. Подключить флаг `-test`.

## Контекст

- Спецификация: [../spec.md](../spec.md), раздел 5 (поведение уведомлений), раздел 2 (флаг `-test`).
- Зависимость: task-02 выполнена (task-03 не обязательна — события можно подавать в тестах напрямую).

## Шаги

1. Добавить `github.com/gen2brain/beeep`.
2. Интерфейс `Notifier` с методом `Show(title, body string) error`; реализация поверх beeep. Интерфейс нужен, чтобы в тестах подменять на фейк и чтобы в будущем заменить бэкенд (см. ADR-002).
3. Тип `Router`: принимает `StateEvent`, находит сущность в конфиге, строит текст по `states` или `template` (`{state}` → значение), вызывает `Notifier`. Состояние без маппинга — лог debug, без уведомления.
4. Троттлинг по разделу 5: на сущность не чаще `min_interval_sec`; схлопнутые события заменяются последним, которое показывается по истечении интервала. Реализация — на таймерах, покрыть тестом с фейковыми часами или короткими интервалами.
5. Флаг `-test` в `main.go`: показать уведомление «Агент работает» / «Уведомления настроены правильно» и выйти с кодом 0.
6. Связать в `main.go`: канал событий из `internal/hass` → `Router`.

## Критерии приёмки

- [ ] `go run ./cmd/agent -test` показывает нативное уведомление на Windows.
- [ ] Юнит-тесты (с фейковым `Notifier`): маппинг `states`; шаблон `template`; состояние без маппинга — уведомления нет; троттлинг схлопывает шквал событий до первого + последнего.
- [ ] Сквозная ручная проверка с реальным HA: смена состояния датчика → нативное уведомление с текстом из конфига.
