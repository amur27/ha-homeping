# Task 03 — WebSocket-клиент Home Assistant

## Цель

Реализовать пакет `internal/hass`: подключение, аутентификация по токену, подписка `subscribe_trigger` на сущности, доставка событий наружу через канал. Переподключение — НЕ в этой задаче (task-05).

## Контекст

- Спецификация: [../spec.md](../spec.md), раздел 4 (протокол), раздел 8 (код выхода 3).
- Протокол HA WebSocket API: сервер шлёт `auth_required` → клиент `auth` → сервер `auth_ok`/`auth_invalid`; далее команды с монотонно растущим `id`.
- Зависимость: task-02 выполнена.

## Шаги

1. Добавить `github.com/coder/websocket`.
2. Тип `Client` с методом `Run(ctx) error`: одно подключение от dial до обрыва; ошибка аутентификации возвращается как выделенная ошибка `ErrAuthInvalid`.
3. Последовательность auth по разделу 4 спецификации; таймаут ожидания каждого ответа — 10 секунд.
4. Подписка `subscribe_trigger` на каждую сущность из конфига; проверять `success: true` в ответе.
5. Приём событий: извлекать `entity_id` и `trigger.to_state.state`, отправлять `StateEvent{EntityID, State}` в канал событий клиента.
6. Ping/pong по разделу 4 (тикер 50 с, ожидание pong 10 с) — отсутствие pong завершает `Run` с ошибкой.
7. В `main.go`: связать клиента с заглушкой-потребителем (пока просто логировать события уровнем info); при `ErrAuthInvalid` — выход с кодом 3.
8. Юнит-тесты протокола через `httptest`-сервер с WebSocket-эндпоинтом, эмулирующим HA: happy path, auth_invalid, отсутствие pong.

## Критерии приёмки

- [ ] Против тестового WS-сервера: аутентификация, подписка и приём события работают (тест зелёный).
- [ ] `auth_invalid` приводит к `ErrAuthInvalid`, процесс завершается кодом 3.
- [ ] Обрыв соединения и молчание на ping завершают `Run` с ошибкой (не паникой, не зависанием).
- [ ] Ручная проверка против реального HA: запустить агента, изменить состояние датчика, увидеть событие в логе.
- [ ] Токен не появляется в логах ни на одном уровне.
