# Task 07 — Супервизор, keyring и логи в файл

## Цель

Подготовить ядро к появлению UI: вынести жизненный цикл клиента HA в супервизор с hot-reload и наблюдаемым статусом, перевести токен на системное хранилище, а логи — в файл. Видимое поведение агента при этом не меняется.

## Контекст

- Спецификация: [../spec.md](../spec.md), разделы 3.3, 9, 10, 11, 12.
- Решение по стеку: [../adr/ADR-003-ui-stack.md](../adr/ADR-003-ui-stack.md).
- Зависимость: task-06 выполнена, v1 работает на обеих машинах.

## Шаги

1. Пакет `internal/agent` — супервизор:
   - владеет клиентом `internal/hass` и конфигом; методы `Run(ctx)`, `Reload()`, `Pause(bool)`, `TestNotification()`;
   - статус (`не настроен | подключение | подключён | отключён | ошибка токена | ошибка конфига | пауза`) — потокобезопасно, с подпиской на изменения (канал/колбэк) для будущих трея и веб-UI;
   - `Reload()` по спеке (раздел 10): невалидный новый конфиг не роняет работающую конфигурацию.
2. Пакет `internal/secrets`: обёртка над `github.com/zalando/go-keyring` (сервис `homeping`, учётная запись `ha_token`) + порядок поиска токена по спеке (раздел 9): keyring → окружение из `token_env`. Ошибку «токен не найден» отличать от прочих ошибок хранилища.
3. `internal/config`: `token_env` сделать необязательным; добавить `logging.file`; функция атомарной записи конфига (временный файл + переименование) — понадобится в task-09.
4. Логирование по спеке (раздел 11): файл по умолчанию для каждой ОС + stderr, ротация при 5 МБ в `.old`.
5. `cmd/agent`: перевести на супервизор; поведение `auth_invalid` и отсутствующего токена по спеке (разделы 4 и 12) — пока без трея агент работает как headless, но ветвление уже по флагу `-no-tray` (сам флаг добавить сейчас, по умолчанию до task-08 он фактически всегда включён).

## Критерии приёмки

- [ ] `go build ./...`, `go vet ./...`, `go test ./...` — чисто.
- [ ] Токен, положенный в Credential Manager (`homeping`/`ha_token`), подхватывается без переменной окружения; при его отсутствии работает fallback на `token_env`.
- [ ] Логи пишутся в `%APPDATA%\homeping\homeping.log`; при превышении 5 МБ создаётся `.old`.
- [ ] `Reload()` с изменённым списком сущностей меняет подписки без перезапуска процесса (проверка вручную: добавить сущность в YAML, вызвать reload, получить уведомление по ней).
- [ ] `Reload()` с невалидным YAML не прерывает работу на прежней конфигурации.
- [ ] Юнит-тесты: порядок источников токена, атомарная запись конфига, смена статусов супервизора.
- [ ] Поведение v1 (уведомления, переподключение, троттлинг) не изменилось.
