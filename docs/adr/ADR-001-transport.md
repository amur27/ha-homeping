# ADR-001: Транспорт доставки событий — WebSocket API Home Assistant

- **Статус**: принято
- **Дата**: 2026-07-02

## Контекст

Нужно доставлять изменения состояний датчиков из Home Assistant (Docker на Raspberry Pi) на два настольных клиента (Windows 11, macOS) в локальной сети. Требования: мгновенность (push, не poll), надёжность, минимум новой инфраструктуры на RPi, работа без облака.

## Рассмотренные варианты

### 1. WebSocket API Home Assistant (выбран)

Штатный API HA (`/api/websocket`): аутентификация long-lived token, команда `subscribe_trigger` даёт push-события по конкретным сущностям.

- ✅ Ноль новых сервисов: API уже работает в существующем контейнере HA.
- ✅ Серверная фильтрация событий (`subscribe_trigger`) — минимальный трафик.
- ✅ Штатная модель безопасности HA: токены создаются и отзываются в профиле.
- ✅ Стабильный, документированный, версионируемый API.
- ❌ Нужно написать клиента — но клиент нужен в любом варианте (см. ниже).

### 2. MQTT (Mosquitto в Docker + подписчики)

- ✅ Классика для IoT, хорошо масштабируется на много подписчиков.
- ❌ Новый контейнер-брокер на RPi: настройка, пользователи/ACL, обновления, мониторинг.
- ❌ HA по умолчанию не публикует состояния в MQTT — нужен `mqtt_statestream` или автоматизации, т.е. настройка ещё и на стороне HA.
- ❌ Клиента на десктопе всё равно нужно писать. Итого: та же работа + лишний брокер.
- Вердикт: оправдан при десятках разнородных подписчиков; для двух клиентов — избыточен.

### 3. ntfy / Gotify (self-hosted push-сервер)

- ✅ Готовая доставка push, красивые мобильные клиенты.
- ❌ Ещё один контейнер + автоматизации в HA для каждой сущности.
- ❌ Официальных **нативных** десктоп-клиентов нет: на Windows/macOS это браузерная вкладка/PWA — уведомления зависят от запущенного браузера и его разрешений, что нарушает требование «системные уведомления».
- Вердикт: хорош для смартфонов, слаб для десктопа.

### 4. Telegram-бот / Pushover и т.п.

- ❌ Облачная зависимость — противоречит локальной архитектуре.
- ❌ Уведомления приходят как сообщения мессенджера, а не системные уведомления о событии дома.
- Вердикт: не соответствует требованиям.

### 5. Companion App

- ❌ Официальные приложения HA существуют только для iOS/Android; поддерживаемого десктопного приложения для Windows/macOS нет.

## Решение

Использовать **WebSocket API Home Assistant** с подпиской `subscribe_trigger` и собственным лёгким агентом на клиентах. Это единственный вариант, дающий push в реальном времени без единого нового сервиса на Raspberry Pi.

## Последствия

- На каждой машине работает фоновый агент (задача task-01…06).
- При выходе за пределы локальной сети архитектура не меняется: тот же агент подключается по `wss://` через reverse proxy.
- Если в будущем подписчиков станет много (например, +умные панели), решение можно пересмотреть в пользу MQTT отдельным ADR.
