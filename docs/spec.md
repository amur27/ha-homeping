# Функциональная спецификация homeping (v2)

Разделы, добавленные или изменённые в v2 относительно v1, помечены **(v2)**.

## 1. Назначение

Фоновый агент, который подключается к WebSocket API Home Assistant, подписывается на изменения состояний перечисленных в конфиге сущностей и показывает нативные уведомления ОС. С v2 агент живёт в трее (Windows) / строке меню (macOS) и настраивается через локальный веб-интерфейс.

## 2. Запуск и параметры командной строки

```
homeping [-config <путь>] [-test] [-version] [-no-tray]
```

- `-config` — путь к YAML-конфигу. По умолчанию:
  - Windows: `%APPDATA%\homeping\config.yaml`
  - macOS: `~/Library/Application Support/homeping/config.yaml`
- `-test` — показать пробное уведомление «Агент работает» и выйти (проверка разрешений ОС).
- `-version` — вывести версию и выйти.
- `-no-tray` **(v2)** — headless-режим без иконки в трее и веб-интерфейса: поведение v1 (для отладки и нестандартных сценариев автозапуска).

**(v2)** Если при запуске конфиг отсутствует, агент не завершается с ошибкой (кроме режима `-no-tray`), а создаёт конфиг-заготовку, стартует в трее со статусом «не настроен» и открывает страницу настроек.

## 3. Конфигурация

### 3.1. Формат файла

```yaml
# Конфигурация агента уведомлений Home Assistant
homeassistant:
  # URL WebSocket API; допустимы схемы ws:// и wss://
  url: "ws://homeassistant.local:8123/api/websocket"
  # (необязательно, v2) Имя переменной окружения с токеном — резервный
  # источник, если токена нет в системном хранилище (см. раздел 9)
  token_env: "HA_TOKEN"

# Сущности, о которых уведомлять
entities:
  - id: binary_sensor.front_door        # entity_id в Home Assistant
    name: "Входная дверь"               # заголовок уведомления
    states:                             # маппинг состояние → текст уведомления
      "on": "🚪 Дверь открыта"
      "off": "Дверь закрыта"
    # Если пришедшее состояние отсутствует в states — уведомление не показывается
  - id: sensor.living_room_temperature
    name: "Температура в гостиной"
    template: "Сейчас {state} °C"       # альтернатива states: шаблон для любого значения

notifications:
  min_interval_sec: 3      # троттлинг: не чаще одного уведомления на сущность за N секунд
  on_disconnect: true      # уведомлять о потере/восстановлении связи с HA

logging:
  level: "info"            # debug | info | warn | error
  # (необязательно, v2) Путь к файлу логов; по умолчанию — см. раздел 11
  file: ""
```

### 3.2. Правила валидации

- `homeassistant.url` обязателен, схема — только `ws`/`wss`.
- **(v2)** `token_env` необязателен; порядок поиска токена — см. раздел 9. Если токен не найден ни в одном источнике: в трей-режиме — статус «не настроен» и открытие настроек, в `-no-tray` — выход с кодом 2.
- `entities` — минимум одна запись; у каждой обязателен `id` и ровно одно из: `states` или `template`.
- Неизвестные ключи в YAML — ошибка (strict-режим), чтобы опечатки не терялись молча.

### 3.3. Запись конфига (v2)

Файл конфига — единственный источник истины; веб-интерфейс — редактор поверх него. Сохранение из UI: валидация → атомарная запись (временный файл + переименование) с сохранением комментариев не требуется → hot-reload (раздел 10). Ручное редактирование YAML по-прежнему поддерживается (пункт меню «Перечитать конфиг»).

## 4. Протокол работы с Home Assistant

1. Открыть WebSocket на `homeassistant.url`.
2. Получить `{"type":"auth_required"}` → отправить `{"type":"auth","access_token":"<токен>"}`.
3. При `auth_invalid` **(v2)**: в трей-режиме — уведомление «Токен отвергнут Home Assistant», статус «ошибка токена», агент продолжает работать и ждёт нового токена из настроек (переподключение не выполняется — с тем же токеном оно бессмысленно); в `-no-tray` — выход с кодом 3, как в v1.
4. При `auth_ok` — для каждой сущности отправить подписку:

```json
{
  "id": <счётчик>,
  "type": "subscribe_trigger",
  "trigger": { "platform": "state", "entity_id": "binary_sensor.front_door" }
}
```

5. На каждое событие `{"type":"event", ...}` извлечь `trigger.to_state.state` и `entity_id`, передать в модуль уведомлений.
6. Раз в 50 секунд отправлять `{"type":"ping"}`; отсутствие `pong` в течение 10 секунд считать обрывом.

## 5. Поведение уведомлений

- Заголовок — `name` сущности; текст — из `states[state]` либо `template` с подстановкой `{state}`.
- Состояние без соответствия в `states` → событие только в лог (debug), без уведомления.
- Троттлинг: события одной сущности чаще `min_interval_sec` схлопываются — показывается только последнее по истечении интервала.
- При `on_disconnect: true`:
  - обрыв связи дольше 30 секунд → одно уведомление «⚠️ Home Assistant недоступен»;
  - восстановление → «✅ Связь с Home Assistant восстановлена».
  Повторные уведомления о том же обрыве не показываются.
- **(v2)** В режиме «Пауза уведомлений» (раздел 7) уведомления не показываются вовсе, включая уведомления об обрыве; события пишутся в лог (debug). Соединение и подписки при этом сохраняются.

## 6. Переподключение

- Экспоненциальный бэкофф: 1, 2, 4, 8, … максимум 60 секунд, плюс джиттер ±20%.
- После успешного переподключения — заново пройти аутентификацию и все подписки.
- Агент не завершается при сетевых ошибках никогда; завершение только по сигналу ОС, из меню трея или по фатальным ошибкам конфигурации/токена (в headless-режиме).

## 7. Трей (v2)

Реализация — `fyne.io/systray`; на macOS агент работает как элемент строки меню (`.app` с `LSUIElement=true`, task-10).

### 7.1. Иконка — индикация статуса

| Статус | Иконка | Когда |
|---|---|---|
| Подключён | обычная | auth_ok, подписки активны |
| Отключён | приглушённая/серая | обрыв связи, идёт бэкофф |
| Пауза | со значком паузы | включена «Пауза уведомлений» |
| Ошибка/не настроен | со значком «!» | auth_invalid, нет токена, невалидный конфиг |

Тултип иконки — текстовый статус («Подключён к Home Assistant» и т.п.).

### 7.2. Меню

| Пункт | Действие |
|---|---|
| `homeping <версия>` | некликабельный заголовок с версией (из git-тега) |
| `Статус: <текст>` | некликабельная строка состояния |
| Настройки… | открыть веб-интерфейс в браузере по умолчанию |
| Тестовое уведомление | показать пробное уведомление |
| Пауза уведомлений | чекбокс; состояние не сохраняется между запусками |
| Открыть конфиг | открыть YAML в приложении по умолчанию |
| Перечитать конфиг | hot-reload из файла (после ручной правки) |
| Выход | корректное завершение (код 0) |

## 8. Веб-интерфейс настроек (v2)

### 8.1. Сервер

- Слушает **только** `127.0.0.1`, порт эфемерный (выделяется ОС при старте).
- Доступ защищён сессионным токеном: генерируется случайно при старте агента, браузер открывается по ссылке `http://127.0.0.1:<порт>/?auth=<токен>`, далее токен переносится в cookie. Запросы без валидного токена — 403.
- Страница — единый HTML/CSS/JS через `go:embed`, без внешних фреймворков и CDN (работает без интернета).

### 8.2. Возможности страницы

- Редактирование URL Home Assistant, параметров `notifications` и `logging.level`.
- Список сущностей: добавление/удаление/правка `id`, `name`, `states`/`template`.
- Ввод токена HA (поле типа password) — сохраняется в системное хранилище, значение никогда не отображается (только признак «токен задан»).
- Статус подключения и кнопки «Проверить подключение», «Тестовое уведомление».
- В подвале страницы — версия агента (из git-тега, та же, что у `-version`).

### 8.3. API

| Метод и путь | Назначение |
|---|---|
| `GET /` | страница настроек |
| `GET /api/status` | статус: состояние подключения, версия, «токен задан: да/нет» |
| `GET /api/config` | текущий конфиг (JSON-представление YAML), без токена |
| `PUT /api/config` | валидация → атомарная запись YAML → hot-reload; при ошибке валидации — 422 с описанием на русском |
| `POST /api/token` | сохранить токен в системное хранилище → hot-reload; тело `{"token":"..."}` |
| `POST /api/test` | показать тестовое уведомление |

Токен HA не возвращается ни одним эндпоинтом и не пишется в лог.

## 9. Хранение токена (v2)

- Основное хранилище — системное: Windows — Credential Manager, macOS — Keychain (`zalando/go-keyring`; сервис `homeping`, учётная запись `ha_token`).
- Порядок поиска при (пере)подключении: 1) системное хранилище → 2) переменная окружения из `token_env` (если ключ задан в конфиге).
- Токен, введённый в веб-интерфейсе, пишется только в системное хранилище.
- Токен никогда: не хранится в YAML, не пишется в логи, не возвращается через API.

## 10. Hot-reload конфига (v2)

- Триггеры: сохранение из веб-UI, пункт меню «Перечитать конфиг», сохранение токена.
- Действия: перечитать и провалидировать конфиг → закрыть текущее соединение с HA → подключиться и подписаться заново с новыми параметрами. Троттлинг-состояние сбрасывается.
- Невалидный конфиг при reload: агент продолжает работать на прежней конфигурации, статус в трее — «ошибка конфига», ошибка видна в веб-UI и логе.
- Слежения за файлом (fsnotify) нет — reload только явный.

## 11. Логирование

- `log/slog`, текстовый формат.
- **(v2)** Основной приёмник — файл (бинарник Windows собирается с `-H=windowsgui`, stdout недоступен):
  - Windows: `%APPDATA%\homeping\homeping.log`
  - macOS: `~/Library/Logs/homeping.log`
  - плюс дублирование в stderr (полезно при запуске из терминала и в `-no-tray`).
- **(v2)** Ротация — простейшая: при достижении 5 МБ файл переименовывается в `homeping.log.old` (одна копия), пишется новый.
- Уровни: подключения и подписки — info; каждое событие — debug; обрывы — warn; фатальное — error.
- В логи **никогда** не пишутся токен HA и сессионный токен веб-UI.

## 12. Коды выхода

| Код | Значение |
|---|---|
| 0 | Штатное завершение по сигналу или из меню трея |
| 2 | Ошибка конфигурации (только `-no-tray`; в трей-режиме — статус «не настроен/ошибка», без выхода) |
| 3 | HA отверг токен (только `-no-tray`; в трей-режиме — статус «ошибка токена», без выхода) |

## 13. Нефункциональные требования

- Память процесса в устойчивом состоянии — не более 40 МБ; размер бинарника — не более ~15 МБ.
- Одна кодовая база; платформенные различия — только внутри `internal/notify`, `internal/tray`, путей по умолчанию и упаковки.
- **(v2)** Сборка: Windows — с основной машины (`CGO_ENABLED=0` больше не гарантируется из-за systray; на Windows cgo не требуется), darwin — на macOS или в CI (cgo для systray), см. task-10.
- `go vet` и `go test ./...` проходят чисто; ключевая логика (парсинг и запись конфига, маппинг состояний, троттлинг, бэкофф, hot-reload, auth веб-UI) покрыта юнит-тестами.
